Git Workflow

Git has three main areas where your files can exist:

  • Working Directory: Where you make changes to your files. This is your local file system.
  • Staging Area (Index): A temporary area where you prepare files before committing. Use git add to move files here.
  • Repository (Local): Where Git permanently stores your committed changes. Use git commit to save staged changes here.

Files flow from Working Directory → Staging Area → Repository. This workflow gives you control over what changes to include in each commit.

Git Ignore

The .gitignore file specifies which files and directories Git should ignore when tracking changes. This is useful for excluding files like build artifacts, dependencies, and sensitive information.

Create a .gitignore file

touch .gitignore

Common patterns to ignore

# Ignore node modules
node_modules/

# Ignore build folders
dist/
build/

# Ignore environment variables
.env
.env.local

# Ignore log files also the file with extensions
*.log

# Ignore OS files
.DS_Store
Thumbs.db

Apply .gitignore to already tracked files

git rm -r --cached .
git add .
git commit -m 'Apply .gitignore'

Reset

Git reset is used to undo commits by moving the HEAD pointer to a previous commit. There are three main types:

1. Soft reset

Moves HEAD to the specified commit but keeps changes in the staging area.Just reset the commit but the changes remains in the stage area. 

git reset --soft head~1

2. Mixed reset

Moves HEAD to the specified commit and unstages changes, but keeps them in the working directory. Just reset the commit and remove the save from the stage area. 

git reset --mixed head~1

Or simply:

git reset head~1

3. Hard reset

Moves HEAD to the specified commit and discards all changes in both staging area and working directory.

git reset --hard head~1

Branches in git

Branches in Git are like separate workspaces where you can work on different features or fixes without affecting the main code.

Why use branches?

  • Work on multiple features at once: You can have one branch for a new feature, another for bug fixes, and keep the main code stable.
  • Safe experimentation: If your changes don't work out, you can just delete the branch without affecting the main code.
  • Team collaboration: Multiple people can work on different branches at the same time without interfering with each other's work.
  • Easy testing: Test new features in a branch before adding them to the main code.

Basic branch commands

Create a new branch

git branch <branch_name>

Switch to a branch

git switch <branch_name>

or

git checkout <branch_name>

Create and switch to a new branch

git switch -c <branch_name>

or

git checkout -b <branch_name>

List all branches

git branch

Delete a branch

git branch -D <branch_name>

Merge a branch into current branch(But be sure you are in main branch and the sub branches must be pushed)

git merge <branch_name>

Git Amend

Git amend is used to modify the most recent commit. It's useful when you want to change the commit message or add forgotten files to the last commit without creating a new commit.

Why use git amend?

  • Fix commit messages: Correct typos or improve the description of your last commit.
  • Add forgotten files: Include files you forgot to stage in the previous commit.
  • Keep history clean: Avoid creating unnecessary commits for small fixes.

Basic amend commands

Amend the last commit message

git commit --amend -m "New commit message"

Amend to add files to the last commit

First, stage the files you want to add:

git add .

Then amend the commit:

git commit --amend --no-edit

The --no-edit flag keeps the same commit message.

Amend and change both files and message

git commit --amend -m "Updated commit message"

Important notes

  • Only amend unpushed commits: If you've already pushed thecommit to a remote repository, amending it will rewrite history and cause issues for others.
  • Force push if necessary: If you must amend a pushed commit, you'll need to force push:
git push --force

However, use this carefully as it can affect your team's work.
